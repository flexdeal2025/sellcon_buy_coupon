> 작성일: 2026-06-24
> 상태: **설계 합의용 (코드 미구현, DB 미적용)** — 셀콘 세션과 규격 합의 후 착수
> 결정 반영: 셀콘은 매입 시점에 코드·유효기간을 **구조화로 보유**(OCR 불필요) / 증빙은 **핵심 세트**

## 0. 한 장 요약

**지금**: 셀콘에서 매입 → 관리자가 이미지 다운로드/코드 정리 → 컨트롤타워 등록화면 업로드 → 검수 → 발행. (수작업 여러 단계)

**목표**: 셀콘이 매입 확정 즉시 컨트롤타워 스테이징에 **자동 푸시** → 검수 → 발행. (업로드·정리 단계 제거)

**핵심 통찰**: 새로 만들 게 아니라 **잇는 것**. 스테이징→검수→발행 파이프라인·안전장치(승인분만 발행·중복 3중 차단)가 이미 있고, 텔레그램 봇이 "외부소스→스테이징" 패턴을 검증했다. 셀콘은 **받는 쪽 API 하나**를 호출하면 된다.

**절대 선**: 자동 푸시는 **스테이징(pending)까지만**. `coupon_codes`(실재고)·발송은 **사람이 발행 버튼**을 눌러야 나간다. 자동 발행 없음.

## 1. 전체 흐름

```
[셀콘 웹앱]  매입 확정
    │  POST /api/stock/ingest  (x-ingest-api-key 인증)
    │  { 상품·코드·유효기간·이미지URL + 핵심 증빙 }
    ▼
[컨트롤타워]  ── 멱등성 검사(source_ref) ── 배치 확보(SC-YYMMDD-NNN)
    │         └→ stock_registrations INSERT (inspection_status='pending')
    ▼
[관리자]  검수(approved)  ──  발행 버튼  ──→  coupon_codes(실재고) / GCP pending(발송대상)
                                  ▲ 사람이 누름(자동화 아님)
```

## 2. API 계약 — `POST /api/stock/ingest` (신규, 컨트롤타워가 제공)

기존 UI용 API(`/api/stock/codes`·`/ocr`)와 **분리된 단일 엔드포인트**를 신설한다. 이유: 배치 자동생성·멱등성·증빙 적재를 한 곳에서 처리하고, 시스템간 인증을 UI 인증과 섞지 않기 위함.

### 2-1. 인증
```
헤더:  x-ingest-api-key: <STOCK_INGEST_KEY>
```
- 컨트롤타워·셀콘 양쪽 `.env`에만 보관(브라우저 번들·코드 하드코딩 금지).
- 기존 `x-app-passcode`(NEXT_PUBLIC_, 브라우저 노출)는 시스템간 호출에 **쓰지 않는다.**

### 2-2. 요청 본문 (코드형/이미지형 공통, `stored_as_code`로 구분)
```jsonc
{
  // ── 멱등성 (필수) ──
  "source": "sellcon",
  "source_ref": "sellcon:payout:<셀콘 정산 UUID>",  // 유니크. 재시도해도 1건만 적재

  // ── 재고 본문 ──
  "stored_as_code": true,                 // true=코드형 / false=이미지형
  "product_name": "스타벅스 아메리카노 T",
  "option_name": "유효기간 최소 10일 이상 쿠폰 발송",
  "coupon_code": "1234-5678-9012",        // 코드형: 필수 / 이미지형: 있으면 포함(파일명 규칙에 필요)
  "expiry_date": "2026-12-31",            // YYYY-MM-DD (셀콘이 구조화 보유)
  "image_url": "https://.../signed?...",  // 이미지형: 필수(서명 URL, TTL ≥ 1시간) / 코드형: 생략

  // ── 핵심 증빙 세트 (필수) ──
  "purchase_channel": "sellcon_auto",
  "proof_type": "no_formal_sellcon",      // 적격증빙 없음(셀콘 정산기록만) — 비중집계 기준
  "payout_uuid": "<셀콘 정산 건 UUID>",    // 모든 증빙의 연결키
  "payout_amount": 27000,                 // 매도자에게 실제 지급한 금액
  "payout_date": "2026-06-24",            // 지급(송금)일
  "seller_name": "홍길동",                 // 매도자 실명
  "purchase_date": "2026-06-24",          // 매입일(지급일과 다를 수 있음)
  "unit_cost": 27000                      // 손익 원가. 셀콘 수수료 있으면 payout_amount와 분리해 명시
}
```

### 2-3. 응답
```jsonc
// 신규 적재
{ "ok": true, "id": "<stock_registrations.id>", "batch_no": "SC-260624-001", "deduped": false }
// 이미 같은 source_ref 존재(재시도) → INSERT 안 하고 기존 행 반환
{ "ok": true, "id": "<기존 id>", "batch_no": "SC-260624-001", "deduped": true }
// 인증 실패
{ "ok": false, "error": "unauthorized" }   // 401
```
- 셀콘은 `deduped:true`도 **성공**으로 처리(재시도 안전).

### 2-4. 이미지 처리 (이미지형)
- 셀콘이 **구조화 데이터 보유** → 타워는 **OCR을 돌리지 않는다.** `image_url`(서명 URL)을 fetch → GCP OCR버킷 업로드 → `image_path` 저장. 상품/코드/유효기간은 셀콘이 보낸 값 사용.
- 검수자는 이미지 원본을 보고 최종 확인(육안 대조).

## 3. 스키마 변경 (컨트롤타워, **적용은 컨펌 후**)

기존 컬럼은 **무수정**, 컬럼만 추가 + 멱등 유니크 인덱스. 파일명(안): `schema_stock_ingest.sql`

```sql
ALTER TABLE public.stock_registrations
  ADD COLUMN IF NOT EXISTS source           text NOT NULL DEFAULT 'manual',
  ADD COLUMN IF NOT EXISTS source_ref       text NOT NULL DEFAULT '',
  -- 핵심 증빙 세트
  ADD COLUMN IF NOT EXISTS purchase_channel text NOT NULL DEFAULT '',
  ADD COLUMN IF NOT EXISTS proof_type       text NOT NULL DEFAULT '',
  ADD COLUMN IF NOT EXISTS payout_uuid      text NOT NULL DEFAULT '',
  ADD COLUMN IF NOT EXISTS payout_amount    integer,
  ADD COLUMN IF NOT EXISTS payout_date      date,
  ADD COLUMN IF NOT EXISTS seller_name      text NOT NULL DEFAULT '';

-- 멱등성: 빈 source_ref(기존 수동등록)는 인덱스에서 제외 → 기존 데이터 안전
CREATE UNIQUE INDEX IF NOT EXISTS idx_stockreg_source_ref
  ON public.stock_registrations (source_ref) WHERE source_ref <> '';
```
기존 수동/텔레그램 등록 행: `source='manual'`, `source_ref=''` → 유니크 인덱스 영향 없음.

## 4. 배치 모델
- 배치번호: **`SC-YYMMDD-NNN`** (SC=셀콘 출처, 매입일 기준, 같은 날 순번). 기존 `YYYYMMDD_NN`·텔레그램 `TG-…`와 구분돼 관리화면에서 출처가 한눈에 보인다.
- 엔드포인트가 해당 날짜 배치 없으면 자동 생성, 있으면 추가.

## 5. 검수·발행 정책 (CS&COO)
- **Phase 1: 전건 `pending`으로 적재 → 사람이 검수·발행.** (가장 안전, 흐름 검증 우선)
- **자동 발행은 절대 금지** — 발행은 알림톡 발송 직결(오배송=환불·신뢰 손상). 발행 버튼은 항상 사람.
- **신속화(자동 승인)는 Phase 5에서**, 아래 선결 후 코드형·고신뢰 건만:
  - 선결① **만료 쿠폰 자동 reject**: 현재 만료건이 경고만 뜨고 등록됨 → 자동승인 도입 전, `expiry < 오늘`이면 `inspection_status='rejected'` 강제.
  - 선결② **중복카운트 컬럼**(`duplicate_count`) 저장: 현재 중복은 메시지로만 소비 → 자동승인 판단에 쓰려면 DB 저장 필요.
  - 자동승인 조건(모두 충족): 코드형 + 만료 D+11↑ + 중복 0 + 코드 비어있지 않음.
  - 예외(저신뢰·임박·중복·고액·이미지형)는 **예외 큐**(관리자 텔레그램 알림)로.

## 8. 부록 — 세무 주의 (운영 정착 시 반영)
- **원천징수 급소**: 동일 매도자에게 **연 300만원 초과** 지급 시 신고의무 발생 가능. `seller_name`/연락처 키로 **매도자별 연 누적**을 집계해 임계 접근 시 알림 권장(Phase 5+). 반복·계속 거래는 사업소득 재분류 위험.
- **이중계상 방지**: 셀콘 매입을 법인카드로 결제했다면 카드장부(`card_transactions_tax`)에도 잡힘 → 원가는 **한 경로로만**. `payout_uuid`로 단방향 연결, 손익에서 카드원가 중복 사용 금지.
- **증빙 연결**: `purchase_proofs`에 `platform='sellcon'` + `payout_uuid` 키로 정산캡처·이체참조를 1:1 보관하면 소명 시 즉시 추출.
- **적격증빙 비중 집계**: `proof_type LIKE 'no_%'` 기준으로 "적격증빙 없는 매입 비중" 상시 쿼리 가능(핵심 세트에 `proof_type` 포함됨).

*— 최종 신고·원천징수 이행은 자문 세무사 확인 권고.*

### 10-2. 확정된 것
- **멱등성 ✅**: 1:1 + `gifticons.id` 안정ID → `source_ref` 깔끔. 재시도 안전.
- **금액 ✅**: `purchasePrice`=원가, `bonusAmount`=별도. 수수료 없음. (이중 칸 분리 전송)
- **트리거 = `APPROVED`**: 이때 코드·유효기간·이미지·매입가 확정 → 재고 빠르게 확보(대표님 '신속' 부합).
- **인프라 ✅**: 셀콘=Vercel, 서버사이드 외부 POST 가능, `.env` 키·`Authorization: Bearer` 지원. ⚠️ Vercel 타임아웃(Pro 60s) → 셀콘은 우리 응답을 오래 기다리지 말고 **fire-and-retry**(202 빠른 응답) 권장.
- **이미지 ✅(MVP)**: 퍼블릭 버킷이라 우리 fetch 즉시 가능. (보안상 서명 URL/RLS는 후속 개선)

| **D1** | 서명 다운로드 URL 미구현(퍼블릭) | MVP는 퍼블릭 URL fetch. 보안 강화 시 셀콘이 `createSignedUrl(TTL)` 추가 | 셀콘(후속) |

### 10-4. 인증 조정
계약 §2-1의 헤더를 셀콘 관례에 맞춰 **`Authorization: Bearer <STOCK_INGEST_KEY>`** 로 변경(셀콘이 이미 이 방식 사용). 의미 동일.

### 10-5. 매도자 신원 — 최소 적재 확정안 (세무 검토 반영)

> 원칙: 동일 법인이라 **셀콘이 매도자 신원의 원본 보관처**(이미 AES 암호화). 타워는 평문 PII를 **복제하지 않고**, 링크+필요시 조회로 대체. 개인정보보호법 안전조치 부담을 셀콘 한 곳으로 집중.

| 구분 | 항목 | 비고 |
|---|---|---|
| **타워에 저장** | `payout_uuid`(=gifticons.id), `seller_ref`(=sellerId, 불투명 링크키), 마스킹명(홍*동), 금액·일자, 상품정보, `proof_type` | 평문 PII 아님 → 타워는 민감도 하위 시스템 유지 |
| **스냅샷 보존(불변)** | 정산완료 시점 1회: `kycName 해시`(평문 아님), `payout_date(확정)`, 금액·`proof_type` | ⚠️ **내구성 안전판** — 셀콘 데이터 소실 시 소명불가→비용 전액 부인+가산세 방지. update 불가 행으로 잠금 |
| **조회로 대체** | 실명·계좌번호·전화·(주민번호) | **소명/원천징수 신고 시에만** `payout_uuid`로 셀콘에서 pull |
| **저장 안 함** | 실명·계좌·전화 **평문** | 복제 금지 |

**원천징수 누적 추적**: 타워에서 `seller_ref`(또는 phoneHash)로 연 누계 집계 → **연 300만원 초과자 추출**(기타소득 원천세 22%·실효 8.8%). 실명은 초과 확정 시점에만 셀콘에서 조회해 신고. ⚠️ **분기/반기 신고 마감 전 자동 집계 루틴** 필요(수작업 누락 방지).

**트리거 모델(확정 2026-06-24) — 셀콘이 보내는 이벤트 3종:**
1. **동의완료(판매 동의 완료, `salesConsentAt`)** → **ingest**(검수대기 재고 생성).
   → 이 시점에 매도자가 셀콘에 **쿠폰 소유권을 이관** = 우리 재고로 전환할 정당한 타이밍.
   → ✅ **데이터 확정 보장**: 셀콘 라이프사이클상 **검수(APPROVED)가 판매 동의보다 항상 먼저** 일어남 → 동의완료 시점엔 코드·유효기간·이미지가 이미 확정(별도 가드 불필요).
   → 순서: **검수완료(데이터확정) → 동의요청 발송 → 동의완료(=발사) → 정산예정(PAYOUT_SCHEDULED)**.
2. `PAYOUT_COMPLETED` → **정산 스냅샷 업데이트**(payout_date 확정 + kycName 해시 + 행 잠금). *Phase 2, 내구성상 중요.*
3. `CANCELLED/REJECTED`(동의완료 후 철회) → **withdraw**(미발행 스테이징행 삭제).

### 10-6. 역방향 동기화 — 셀콘 '재고 전환 완료' 자동 태깅
현재 셀콘 어드민 '매입 현황'에서 관리자가 **수동**으로 '재고 전환 완료' 태그를 찍는다. 이를 자동화한다.

**핵심: 별도 콜백/웹훅 불필요. 우리 API의 성공 응답이 곧 확인서다.**
- 셀콘이 `POST /api/stock/ingest` 호출 → 우리가 검수대기 행 생성 후 **`{ ok:true, id }`** 반환.
- 셀콘은 그 **성공 응답을 받으면 스스로** '재고 전환 완료' 태그 + 우리 `id`(`towerStockId`)를 자기 DB에 기록.
- 멱등 재시도로 `{ deduped:true, id }`를 받아도 동일하게 태깅(이미 전환된 것).
- **자동 전환(동의완료 훅)과 수동 전환(어드민 예외처리)이 같은 API를 거치도록** → 태그가 한 경로로 일원화. 어느 쪽이든 ingest 성공 시 태깅.
- 실패(ok 미수신) 시 태그 안 찍힘 → 셀콘이 재시도 또는 어드민에 '전환 실패' 노출. **"태그 없음 = 미전환"** 이 명확히 구분됨.
- 담당: 태그 기록은 **셀콘 측**(우리 응답을 보고 셀콘이 자기 매입현황에 반영). 우리는 **명확한 응답(ok·id·deduped)** 만 보장.
